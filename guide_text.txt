1. Project Overview & Goals
Cerebus is a Dual LLM Guardrail Architecture that places a dedicated Security LLM as a gatekeeper in front of a primary Core LLM. Every user prompt is screened for prompt injections, jailbreak attempts, and sensitive data extraction before being allowed to reach the main model. Outputs from the Core LLM are then re-checked by an Output Guardrail before reaching the user.
Why Cerebus?
Traditional guardrails rely on static keyword lists and rule-based filters — both brittle against novel attack patterns. Cerebus uses AI to guard AI: a secondary model with adversarial awareness dynamically evaluates each request in context, enabling real-time, adaptive protection.
1.1  Core Goals
Block prompt injection, jailbreak, and sensitive data extraction attacks in real time
Provide context-aware input AND output validation (dual guardrail)
Be deployable as a microservice wrapper around any existing LLM
Support enterprise-grade observability via security logs
Keep latency overhead under 300 ms per request
1.2  Architecture Summary
The pipeline flows through five logical stages:
User sends a prompt to the Orchestrator
Input Guardrail (DistilBERT-based) screens for prompt injection, jailbreak, and PII/sensitive data
If SAFE → forwarded to Primary LLM (with optional RAG Knowledge Base)
Output Guardrail (lightweight LLM) screens response for toxicity, bias, hallucination, and data leaks
If VALID → Final response delivered; otherwise → Block / Policy Message sent
2. Concepts to Study (Pre-Implementation)
Before writing a single line of code, make sure you understand the following topics. Estimated self-study time: 2–3 days for someone with ML basics.
2.1  LLM Security Attacks
Attack Type
What You Need to Know
Prompt Injection
Attacker embeds instructions inside user data to override system prompts. Study direct vs indirect injection.
Jailbreak Attacks
Social-engineering patterns (DAN, role-play, base64 encoding) that bypass safety filters.
Sensitive Data Extraction
Prompts designed to leak training data, API keys, PII, or system-prompt contents.
Indirect Injection
Malicious instructions hidden inside documents, web pages, or tool outputs retrieved by the LLM.
Model Inversion
Inferring training data from repeated model queries — relevant for output validation.

Phase 2 — Input Guardrail (Days 2–3)
The Input Guardrail is the most critical component. It classifies each incoming prompt into one of four labels: SAFE, PROMPT_INJECTION, JAILBREAK, SENSITIVE_EXTRACTION.
Step 2A — Build the Dataset
Download JailbreakBench and Lakera PINT datasets
Create a balanced CSV with columns: text, label (0=SAFE, 1=INJECTION, 2=JAILBREAK, 3=SENSITIVE)
Add at least 200 SAFE examples (normal user queries) to avoid false positives
Step 2B — Fine-tune DistilBERT with LoRA

    task_type=TaskType.SEQ_CLS, r=8, lora_alpha=32,

Step 2C — Build the FastAPI Service
# input_guardrail/main.py
from fastapi import FastAPI
from pydantic import BaseModel
app = FastAPI()
class PromptRequest(BaseModel):
    prompt: str
    session_id: str
@app.post('/analyze')
async def analyze(req: PromptRequest):
    label, confidence = classifier(req.prompt)
    return {'safe': label == 'SAFE', 'label': label, 'confidence': confidence}
Phase 3 — Orchestrator (Day 4)
The Orchestrator is the single entry point. It receives user requests, calls the Input Guardrail, routes to Core LLM if safe, and calls the Output Guardrail before responding.
# orchestrator/main.py — simplified flow
@app.post('/chat')
async def chat(req: ChatRequest):
    # Step 1: Input Guardrail check
    guard_result = await http_client.post(INPUT_GUARDRAIL_URL, json={'prompt': req.prompt})
    if not guard_result['safe']:
        log_security_event(req, guard_result)
        return {'response': 'Request blocked by security policy.', 'blocked': True}
    # Step 2: Core LLM
    llm_response = await call_core_llm(req.prompt)
    # Step 3: Output Guardrail check
    out_check = await http_client.post(OUTPUT_GUARDRAIL_URL, json={'response': llm_response})
    if not out_check['valid']:
        return {'response': 'Response filtered by safety policy.', 'blocked': True}
    return {'response': llm_response, 'blocked': False}
Phase 4 — Output Guardrail (Day 5)
The Output Guardrail runs four parallel checks on the Core LLM response:
Detector
Implementation Approach
Toxicity Detector
Use the Detoxify library (pip install detoxify) — free, open-source
Bias Detector
Fine-tune a classifier on CrowS-Pairs dataset or use HF 'valurank/distilroberta-base-offensive-value-detector'
Hallucination Detector
Compare response claims against RAG source docs using NLI (facebook/bart-large-mnli)

Regex + NER (spaCy) to detect PII: email, phone, SSN, credit card patterns
Phase 5 — RAG Knowledge Base (Day 6)
Prepare your domain-specific documents (PDFs, text files)
Chunk documents into 512-token segments with 50-token overlap
Generate embeddings using sentence-transformers (all-MiniLM-L6-v2 — free & fast)
Store vectors in FAISS index (runs entirely locally, no API cost)
At query time, retrieve top-K relevant chunks and inject as context into the Core LLM prompt
from langchain_community.vectorstores import FAISS
from langchain_huggingface import HuggingFaceEmbeddings
embeddings = HuggingFaceEmbeddings(model_name='all-MiniLM-L6-v2')
vectorstore = FAISS.from_documents(chunks, embeddings)
retriever = vectorstore.as_retriever(search_kwargs={'k': 4})
Phase 6 — Security Logging & Redis (Day 7)
Use Redis for rate limiting: max 10 requests/minute per session_id using Redis sliding window counters
Log all blocked requests to SQLite (development) or PostgreSQL (production) with: timestamp, session_id, prompt_hash (not plaintext), label, confidence, action_taken
Expose a /logs endpoint (admin-only, API-key protected) for audit dashboards
Optionally hook Prometheus metrics: requests_total, blocked_total, latency_histogram

6. Likely Judging Questions & Answers
Tip for Presentation
Judges at hackathons typically probe on three axes: technical depth, novelty vs existing work, and scalability. Prepare concise 2-sentence answers to each question below and practice them out loud.
Q1: Why not just use Llama Guard or OpenAI Moderation API?
These are single-layer, input-only solutions. Cerebus adds an Output Guardrail layer and can be fine-tuned on domain-specific adversarial datasets — e.g., medical or financial prompt patterns — whereas Llama Guard is a fixed general-purpose classifier. Cerebus also logs all blocked attempts for audit trails, which enterprise deployments require.
Q2: What is the latency overhead of adding two extra LLM calls?
DistilBERT inference on CPU runs in ~30–50 ms per request. The Output Guardrail using Flan-T5-Base adds ~80–120 ms. Total overhead is under 200 ms in 95th percentile, which is acceptable for most enterprise applications. Redis caching of seen-before prompt patterns can further reduce input guardrail latency to near zero for repeated prompts.
Q3: How do you handle false positives — blocking legitimate prompts?
The fine-tuned classifier outputs a confidence score. We set a threshold (e.g., confidence > 0.85) before blocking. Prompts in the 0.70–0.85 range are flagged for human review rather than hard-blocked. The SAFE class is deliberately over-represented (60%) in training data to minimise false positives. Feedback from flagged prompts is used to continuously retrain the guardrail model.
Q4: Can this be bypassed with adversarial suffixes or encoded attacks?
No system is 100% bypass-proof. Cerebus addresses this by training on adversarial datasets that include Base64, Pig Latin, and token-smuggling examples. The Output Guardrail provides a second opportunity to catch attacks that slip through the input layer. We also log all borderline cases for model improvement — this is an iterative defence, not a one-shot silver bullet.
Q5: How does the RAG Knowledge Base interact with the security system?
The RAG Knowledge Base is accessed only after the Input Guardrail passes the prompt as safe. Retrieved chunks are injected into the Core LLM's context. The Output Guardrail then uses those same retrieved chunks as ground truth to perform hallucination detection — if the model's response contradicts the source documents, it is flagged and blocked.
